# Backtesting Module Specification

**Module:** Backtesting  
**Files:** `train_robot.py`, `data.py` (SimulationData class)  
**Purpose:** Simulate trading strategy performance on historical data, executing buy/sell decisions, managing positions, and accumulating performance metrics.

---

## 1. Overview

The Backtesting Module simulates how trading strategies would have performed in the past using historical OHLC data and signals. It executes the full trading lifecycle—from signal evaluation to position entry/exit, stop-loss management, and profit/loss calculation—without actual market interaction. This enables fast, repeatable evaluation of strategies before live deployment.

The module's core component, TrainRobot, replays historical data point-by-point, calling strategy checks at each timestamp. The robot maintains position state (open, closing, closed), executes entry/exit actions when signals trigger, and tracks cumulative revenue. To accelerate backtest on multi-core machines, the Trainer orchestrates parallel TrainRobot instances, each processing a time segment independently, then aggregates results.

Key design principle: TrainRobot mirrors Robot's interface but operates on static historical data instead of live market feeds, ensuring backtest behavior matches live trading behavior.

---

## 2. Module Boundaries

### Upstream Dependencies
- **Data Module:** `data.py` provides OHLC candles and indicators; `SimulationData` loads `df_with_indicators.pkl` once and yields `WideDataPoint` at each step
- **Training Module:** `trainer.py` generates data points; orchestrates parallel simulation
- **Signals & Strategy:** Strategy Manager evaluates signal conditions and generates trading actions
- **Position Module:** Position class manages entry/exit logic, fee handling, PnL calculation
- **Environment Configuration:** Data path, time range, and thread count from `os.environ`

### Downstream Consumers
- **Trainer:** Aggregates results from parallel backtest processes
- **Performance Analyzer:** Consumes backtest results for statistics and visualization
- **Visualization:** Charts and dashboards display backtest equity curves and trade-level details

### Key Interfaces Exposed
- **`TrainRobot.__init__(thread_num=0)`** — Initialize backtest robot for a specific thread
- **`TrainRobot.step(data_point: WideDataPoint, action)`** — Execute one simulation step: evaluate strategy, execute action, update position; replaces the old `set_data(cur_data)` + `do()` pair
- **`TrainRobot.get_results(thread_id) -> dict`** — Return accumulated revenue and trade detail after simulation ends
- **`SimulationData.get() -> WideDataPoint`** / **`SimulationData.next()`** / **`SimulationData.is_end() -> bool`** — Replay control

## 4. Execution Flow

### A. TrainRobot.step(data_point, action) — Single Step

Executes one simulation iteration. Called once per replay timestamp by the `trainer.py` loop:

```python
sim = SimulationData(pair, begin_ts, end_ts, step_min)
while not sim.is_end():
    data_point = sim.get()
    action = strategy_manager.check(data_point, ...)
    train_robot.step(data_point, action)
    sim.next()
```

1. **Log Current Position State:**
   - Output position state (open/closed, entry price, etc.)

2. **Check Position Action Queue:**
   - If position has a queued action (open, close, safety): route to `buy(data_point)` or `sell(data_point)` as appropriate

3. **Execute Strategy Action (`action` param):**
   - Call `wait(action, data_point)` to handle strategy_action and manage transitions

4. **Exception Handling:**
   - Catch exceptions, log with timestamp, continue to next step

### B. TrainRobot.buy() Execution

Executes buy action (entry long or exit short):

1. **Get Current Low Price:**
   - Extract from latest candle

2. **Check Safety Buy (Exit Short):**
   - If waiting for buy (short position closing):
     - If price <= target close: execute buy at close price
     - Record "safety buy" in logs

3. **Check Stop Loss (Short Position):**
   - Delegated to `position.is_stop_loss_triggered(cur_price)` — no inline stop-loss check in TrainRobot
   - If triggered: execute buy at stop_loss price; record "STOP LOSS" in logs

4. **Check Entry (Open Long):**
   - If action == OPEN_LONG:
     - Validate entry conditions via `position.check_stop_open()`
     - Check for time-based exit via `position.close_by_time()`
     - If price <= target open: execute buy

5. **Process Buy:**
   - Calculate amount: `coin_amount = position.get_action_amount(price) / price`
   - Call `position.record_entry_fill(coin_amount, price)`

6. **Update Position State:**
   - If position now closed (short exit): finalize, accumulate revenue
   - Else: transition to next state (safety wait)

### C. TrainRobot.sell() Execution

Executes sell action (exit long or entry short):

1. **Get Current High Price:**
   - Extract from latest candle

2. **Check Safety Sell (Exit Long):**
   - If waiting for sell (long position closing):
     - If price >= target close: execute sell at close price

3. **Check Stop Loss (Long Position):**
   - Delegated to `position.is_stop_loss_triggered(cur_price)` — no inline stop-loss check in TrainRobot
   - If triggered: execute sell at stop_loss price

4. **Check Entry (Open Short):**
   - If action == OPEN_SHORT:
     - Validate via `position.check_stop_open()`
     - If price >= target open: execute sell

5. **Process Sell:**
   - Calculate amount: `coin_amount = position.get_action_amount(price)`
   - Call `position.record_exit_fill(coin_amount, price)`

6. **Update Position State:**
   - If position now closed (long exit): finalize, accumulate revenue

### D. TrainRobot.wait() State Management

Handles deferred actions and position transitions:

1. **Check Stop Loss Setup:**
   - If position needs stop-loss: call `position.setup_stop_loss()`
   - If setup fails: assert (shouldn't happen in backtest)
   - Record stop-loss event in action message
   - Execute exit: call `sell()` for long, `buy()` for short

2. **Safety Update (Position Protection):**
   - Call `position.safety_update()` to check for protective exits
   - May override strategy action if safety criteria met

3. **Handle Entry Actions (OPEN_LONG / OPEN_SHORT):**
   - Call `position.open()` with strategy parameters
   - If successful: record "OPEN LONG/SHORT" event
   - Execute fill: call `buy()` for long, `sell()` for short

4. **Handle Exit Actions (CLOSE_LONG / CLOSE_SHORT):**
   - Call `position.close()` with strategy parameters
   - If successful: record "CLOSE LONG/SHORT" event
   - Exit: call `sell()` for long, `buy()` for short

## 5. Key Components

### TrainRobot Class
Main simulator class. Manages position state, execution simulation, and result accumulation.

**Key attributes:**
- `position` — Position tracker (open price, quantity, fees, PnL)
- `strategy_mngr` — Strategy manager for signal evaluation
- `thread_num` — Thread ID for parallel execution tracking
- `total_revenue` — Cumulative revenue (relative: realized_pnl / initial_investment)
- `total_revenue_abs` — Absolute revenue (USD equivalent)
- `total_revenue_by_id` — Revenue breakdown by strategy

**Note:** `data` is not stored as an attribute — `data_point` is passed per-call to `step()`.

### SimulationData Class
Loads `df_with_indicators.pkl` once at init; yields `WideDataPoint` at each replay step.

**Methods:**
- `__init__(pair, begin_ts, end_ts, step_min)` — Load wide DataFrame; build timestamp range
- `get() -> WideDataPoint` — Current replay point (O(1), no I/O)
- `next()` — Advance to next timestamp
- `is_end() -> bool` — True when replay is complete
- `steps -> int` — Total replay length

### Position & Action Classes
(Defined in position module; used by TrainRobot)

- Position: Tracks entry/exit prices, quantity, fees, PnL
- Action: Records entry/exit events with timestamps and prices

## 6. Existing Approach

### Simulation Loop Pattern

The `trainer.py` simulation loop calls `train_robot.step(data_point, action)` once per timestamp:
```python
sim = SimulationData(pair, begin_ts, end_ts, step_min)
while not sim.is_end():
    data_point = sim.get()
    action = strategy_manager.check(data_point, ...)
    train_robot.step(data_point, action)
    sim.next()
```

`step()` replaces the old `set_data(cur_data)` + `do()` pair — data is passed in rather than stored.

### Position-Driven State Machine

Position state drives execution flow:
- POSITION_WAIT_BUY: Waiting for buy signal
- POSITION_WAIT_SELL: Waiting for sell signal
- POSITION_OPEN: Trade active
- POSITION_CLOSED: Trade complete

TrainRobot checks state and routes to appropriate buy/sell/wait handler.

### Thread-Based Parallelization

Time range split across processes. Each TrainRobot instance processes independent segment:
```python
# Trainer orchestrates:
for time_segment in split_time_range(num_threads):
    p = Process(target=parallel_simulate, args=(segment,))
    processes.append(p)
```

Results aggregated on main thread.

### Multi-Timeframe Signal Aggregation

Strategy manager aggregates signals across timeframes (1m, 5m, 15m, 60m, 240m, 1440m — from `candles_config.yaml`). Data is pre-computed in `df_with_indicators.pkl`; `WideDataPoint.get(col, tf, shift)` provides path-agnostic access.

### Revenue Tracking (Relative & Absolute)

- **Relative:** `total_revenue = (realized_pnl) / (initial_investment)` as percentage
- **Absolute:** `total_revenue_abs` = fees-inclusive USD equivalent

Both tracked to validate strategy performance under different scaling.

## 7. Potential Improvements

1. **Add Walk-Forward Optimization**
   - Divide backtest period into overlapping windows
   - Train on one window, test on next (prevents overfitting)
   - Would provide more realistic performance estimates

2. **Implement Slippage Simulation**
   - Add random price deviation at execution (market impact)
   - Simulate partial fill scenarios
   - Would reflect real-world execution costs

3. **Add Liquidity Constraints**
   - Limit order size based on historical volume
   - Reject orders if volume insufficient
   - Would prevent phantom fills on low-liquidity instruments

4. **Implement Partial Position Sizing**
   - Start with small position, scale up on confirmation
   - Reduce position on profit targets
   - Would improve risk management simulation

5. **Add Correlation Analysis**
   - Track correlation between backtest trades and market conditions
   - Identify regime-dependent strategy performance
   - Would improve strategy robustness assessment

6. **Implement Monte Carlo Simulation**
   - Resample historical trades randomly
   - Generate distribution of possible outcomes
   - Would provide confidence intervals on performance

7. **Add Equity Curve Statistics**
   - Calculate Sharpe ratio, max drawdown, win rate per backtest
   - Export to standard performance metrics
   - Would enable systematic strategy ranking

8. **Support Intrabar Execution**
   - Simulate fills at multiple prices within same candle
   - Match limit orders against intracandle price extrema
   - Would improve realism on fast-moving markets

9. **Add Commission Optimization**
   - Test different fee structures in simulation
   - Identify fee sensitivity
   - Would help evaluate broker selection impact

10. **Implement Event-Driven Architecture**
    - Separate data loading from trading logic
    - Allow pluggable position managers and order processors
    - Would improve modularity and testability
